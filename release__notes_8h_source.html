125 /** @page providers_changes Changes to ChannelProvider ownership rules
126 
127 @tableofcontents
128 
129 Major series 6.x includes changes to the rules for when user code may
130 store strong and/or weak references to epics::pvAccess::ChannelProvider
131 and related classes. These rules exist to prevent strong reference loops from forming.
132 
133 @section providers_changes_requester Operation <-> Requester
134 
135 One change is the reversal of the allowed relationship between
136 an Operation and its associated Requester (see @ref provider_roles_requester for definitions).
137 
138 Prior to 6.0.0 an Operation was required to hold a strong reference to its Requester.
139 This prevented the Requester from holding a strong ref to the Operation.
140 
141 This was found to be inconvienent and frequently violated.
142 Beginning with 6.0.0 an Operation is prohibited from holding a strong reference to its Requester.
143 
144 @subsection providers_changes_requester_port Porting
145 
146 Legecy code does not store a strong reference to Requesters will see that they are immediately destory.
147 
148 An example where the Operation is a ChannelGet and the Requester is ChannelGetRequester.
149 
150 @code
151 // Bad example!
152 epics::pvAccess::Channel::shared_pointer chan = ...;
153 epics::pvAccess::ChannelGet::shared_pointer get;
154 {
155  epics::pvAccess::ChannelGetRequester::shared_pointer req(new MyRequester(...));
156  get = chan->createChannelGet(req, epics::pvData::createRequest("field()"));
157  // 'req' is only strong ref.
158  // MyRequester::~MyRequester() called here
159  // MyRequester::getDone() never called!
160 }
161 @endcode
162 
163 It is necessary to maintain a strong reference to the ChannelRequester as long as callbacks are desired.
164 
165 @note Legacy code could be modified to strong each Requester alongside the associated Operation.
166 
167 New code may utilize the new ownership rules and store the Operation in a custom Requester.
168 
169 @code
170 struct MyRequester : public epics::pvAccess::ChannelGetRequester {
171  epics::pvAccess::ChannelGet::shared_pointer get;
172  ...
173 };
174 epics::pvAccess::ChannelGetRequester::shared_pointer req(new MyRequester(...));
175 epics::pvAccess::Channel::shared_pointer chan = ...;
176 req->get = chan->createChannelGet(req, epics::pvData::createRequest("field()"));
177 @endcode
178 
179 @section providers_changes_store Must store Operation reference
180 
181 Beginning with 6.0.0, all create methods of
182 epics::pvAccess::ChannelProvider and
183 epics::pvAccess::Channel
184 are required to return shared_ptr which are uniquely ownered by the caller
185 and are not stored internally.
186 
187 Thus the caller of a create method must keep a reference to each Operation
188 or the Operation will be destoryed.
189 
190 Prior to 6.0.0 some providers, notibly the "pva" provider, did not do this.
191 It was thus (sometimes) necessary to explicitly call a destory() method
192 to fully dispose of an Operation.
193 Failure to do this resulted in a slow resource leak.
194 
195 Beginning with 6.0.0 an Operation must not rely on user code to call a destory() method.
196 
197 @subsection providers_changes_store_port Porting
198 
199 Legecy code may be relying on these internal references to keep an Operation alive.
200 
201 Beginning with 6.0.0 the shared_ptr returned by any create method must be stored or the Operation will
202 be immediately destroy'd.
203 
204 @note Beginning with 6.0.0 use of the epics::pvAccess::ChannelRequester interface is optional
205  and may be omitted when calling createChannel().
