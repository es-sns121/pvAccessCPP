18 @section providers_roles Roles
19 
20 The Client and Server APIs revolve around the epics::pvAccess::ChannelProvider class.
21 
22 In the following discussion the @ref providers_client calls methods of ChannelProvider and associated classes.
23 The @ref providers_server implements ChannelProvider and associated classes and is called by a client.
24 
25 By convention, instances of ChannelProvider are registered and retrieved through one of
26 epics::pvAccess::ChannelProviderRegistry::clients()
27 or
28 epics::pvAccess::ChannelProviderRegistry::servers()

30 @subsection provider_roles_requester Operation and Requester
31 
32 The classes associated with ChannelProvider come in pairs.
33 eg.
34 
35 - epics::pvAccess::Channel and epics::pvAccess::ChannelRequester
36 - epics::pvAccess::ChannelGet and epics::pvAccess::ChannelGetRequester
37 - epics::pvAccess::ChannelPut and epics::pvAccess::ChannelPutRequester
38 - epics::pvAccess::Monitor and epics::pvAccess::MonitorRequester
39 - epics::pvAccess::ChannelRPC and epics::pvAccess::ChannelRPCRequester
40 - epics::pvAccess::ChannelProcess and epics::pvAccess::ChannelProcessRequester
41 - epics::pvAccess::ChannelPutGet and epics::pvAccess::ChannelPutGetRequester
42 - epics::pvAccess::ChannelArray and epics::pvAccess::ChannelArrayRequester
43 
44 In the following discussions the term "Operation" refers to eg. Channel, ChannelGet, or similar
45 while "Requester" refers to ChannelRequester, ChannelGetRequester, or similar.
46 
47 The "Requester" classes are implemented by the Client role
48 and called by the Server role to give notification to the client of certain events.
49 For example,
50 epics::pvAccess::ChannelRequester::channelStateChange()
51 is called when a Channel becomes (dis)connected.
52 
53 A "Requester" sub-class must be provided when each "Operation" is created.
54 This Requester then becomes bound to the Operation.
55 
56 @note An exception to this is epics::pvAccess::ChannelProvider::createChannel()
57  Where a epics::pvAccess::ChannelRequester may be omitted.
58 
59 For convenience each Operation class has a member typedef for it's associated Requester, and vis. versa.
60 For example ChannelGet::requester_type is ChannelGetRequester
61 and ChannelGetRequester::operation_type is ChannelGet.
62 
63 @subsubsection provider_roles_requester_locking
64 
65 Operations methods may call requester methods, and vis versa.
66 The following rules must be followed to avoid deadlocks.
67 
68 - No locks must be held when Requester methods are called.
69 - Locks may be held when Operation methods are called.
70 
71 These rules place the burdon of avoiding deadlocks on the ChannelProvider implementation (Server role).
72 
73 Clients must still be aware when some Operation methods can call some Requester methods recursively,
74 and consider this when locking.
75 
76 For example, the following call stack may legitimetly occur for a ChannelProvider
77 to for a Get which accesses locally stored data.
78 
79 - Channel::createChannelGet()
80  - ChannelGetRequester::channelGetConnect()
81  - ChannelGet::get()
82  - ChannelGetRequester::getDone()
83 
84 Thus care should be taken when calling ChannelGet::get() from within ChannelGetRequester::getDone()
85 to avoid infinite recursion.

87 @subsection providers_ownership shared_ptr<> and Ownership
88 
89 "Operations" and "Requesters" are always handled via std::tr1::shared_ptr.
90 
91 In the following dicussions an instance of std::tr1::shared_ptr is referred to as a "reference",
92 specifically a strong reference.
93 The associated std::tr1::weak_ptr is referred to as a weak reference.
94 
95 shared_ptr instances can exist on the stack (local variables) or as
96 struct/class members.
97 
98 Situations where an object contains a reference to itself, either directly or indirectly,
99 are known as "reference loops".
100 As long as a reference loop persists, any cleanup of resources associated with
101 the shared_ptr (s) involved will not be done.
102 A reference loop which is never broken is called a "reference leak".
103 
104 In order to avoid reference leaks, required relationships between
105 various classes will be described, and some rules stated.
106 
107 In discussing the usage of an API diagrams like the following will be used to illustrate roles and ownership requirements.
108 
109 The distinction of what is "user" code will depend on context.
110 For example, when discussing the Client role, epics::pvAccess::Channel will not be implemented by "user code".
111 When discussing the Server role, user code will implement Channel.
112 
113 @subsubsection providers_ownership_unique Uniqueness
114 
115 A shared_ptr is said to be unique if it is the only strong reference to the underlying object.
116 In this case shared_ptr::unique() returns true.
117 
118 The general rule is that functions which create/allocate new objects using shared_ptr must yield a unique shared_ptr.
119 Yielding a non-unique shared_ptr is a sign that an internal reference leak exists.
120

124 - An oval denotes a class not implemented by user code

149 @section providers_client Client Role
150 
151 A client will by configured with a certain provider name string.
152 It will begin by passing this string to
153 epics::pvAccess::ChannelProviderRegistry::createProvider()
154 to obtain a ChannelProvider instance.
155 
156 Custom configuration of provider can be accomplished by
157 passing a
158 epics::pvAccess::Configuration
159 to createProvider().
160 A Configuration is a set of string key/value parameters
161 which the provider may use as it sees fit.
162

165 By convention default Configuration use the process environment.

180 @subsection providers_client_channel Client Channel
181 
182 The primary (and in many cases only) ChannelProvider method of interest is
183 epics::pvAccess::ChannelProvider::createChannel()
184 from which a new
185 epics::pvAccess::Channel
186 can be obtained.
187 
188 Each call to createChannel() produces a "new" std::shared_ptr<Channel>
189 which is uniquely owned by the caller (see @ref providers_ownership_unique).
190 As such, the caller must keep a reference to to the Channel or it will be destroyed.
191 This may be done explicitly, or implicitly by storing a reference to an Operation.
192 
193 @note The returned Channel does *not* hold a strong reference for the ChannelProvider
194  from which it was created.
195  User code *must* keep a reference to the provider as long as Channels are in use.
196  All Channels are automatically closed when their provider is destroyed.
197 
198 A Channel can be created at any time, and shall succeed as long as
199 the provided name and address are syntactically valid, and the priority is in the valid range.
200 
201 When created, a Channel may or may not already be in the Connected state.
202 
203 On creation epics::pvAccess::ChannelRequester::channelCreated will be called
204 before createChannel() returns.
205 
206 Notification of connection state changes are made through
207 epics::pvAccess::ChannelRequester::channelStateChange()
208 as well as through the *Connect() and channelDisconnect() methods
209 of Requesters of any Operations on a Channel (eg.
210 epics::pvAccess::MonitorRequester::channelDisconnect()
211 ).

213 @subsection providers_client_operations Client Operations
214 
215 This section describes commonalities between the various Operation supported:
216 Get, Put, Monitor, RPC, PutGet, Process, and Array.
217 
218 An Operation is created/allocated with one of the create* methods of Channel.
219 All behave in a similar way.
220 
221 - epics::pvAccess::Channel::createChannelGet()
222 - epics::pvAccess::Channel::createChannelPut()
223 - epics::pvAccess::Channel::createMonitor()
224 - epics::pvAccess::Channel::createChannelRPC()
225 - epics::pvAccess::Channel::createChannelPutGet()
226 - epics::pvAccess::Channel::createChannelProcess()
227 - epics::pvAccess::Channel::createChannelArray()
228 
229 The created Operation is unique (see @ref providers_ownership_unique).
230 
231 The \*Connect() method of the corresponding Requester will be called when
232 the Operation is "ready" (underlying Channel is connected).
233 This may happen before the create* method has returned, or at some time later.
234 
235 - epics::pvAccess::ChannelGetRequester::channelGetConnect()
236 - epics::pvAccess::ChannelPutRequester::channelPutConnect()
237 - epics::pvAccess::MonitorRequester::monitorConnect()

240 - epics::pvAccess::ChannelProcessRequester::channelProcessConnect()
241 - epics::pvAccess::ChannelArrayRequester::channelArrayConnect()

288 @subsubsection providers_client_operations_monitor Monitor Operation
289 
290 epics::pvAccess::Monitor operations are handled differently than others as
291 more than one subscription update may be delivered after start() is called.
292 
293 During or after epics::pvAccess::MonitorRequester::monitorConnect()
294 it is necessary to call epics::pvAccess::Monitor::start()
295 to begin receiving subscription updates.
296 
297 The epics::pvAccess::Monitor::poll() and epics::pvAccess::Monitor::release()
298 methods access a FIFO queue of subscription updates which have been received.
299 The epics::pvAccess::MonitorRequester::monitorEvent() method is called
300 when this FIFO becomes not empty.
301 
302 @note The pvAccess::MonitorRequester::monitorEvent() is called from a server internal thread
303  which may be shared with other operations.
304  In order to avoid delaying other channels/operations
305  it is recommended to use monitorEvent() as notification for a client
306  specific worker thread where poll() and release() are called.
307 
308 epics::pvAccess::MonitorRequester::unlisten() is called to indicate a subscription
309 has reached a definite end without an error.
310 Not all subscription sources will use this.
311 
312 @warning It is critical that any non-NULL MonitorElement returned by poll()
313  must be passed to release().
314  Failure to do this will result in a resource leak and possibly stall
315  the monitor.
316  See epics::pvAccess::Monitor::Stats::noutstanding
317 
318 epics::pvAccess::Monitor::getStats() can help diagnose problems related
319 to the Monitor FIFO.
320 See epics::pvAccess::Monitor::Stats.
